01

System Overview

Four deployable services share one application database, with a separate database backing Keycloak. Two client apps — the customer mobile app and the vendor web portal — talk to a single backend API.

A · API

Backend

Express + Prisma. Catalog, orders, auth, vendor/admin, and a WebSocket for live order tracking.

B · WEB

Vendor Portal

Next.js 15 single-page app: onboarding, login, dashboard, product editing, order & admin views.

C · APP

Mobile

Expo / React Native (also web-exported). 11 screens, OTP or Keycloak login, live tracking.

D · IDP

Keycloak

Customer identity provider (production target). Realm bazzato imported on boot.

Key principle Two independent auth systems. Customers authenticate via Keycloak (or mock OTP in dev). Vendors and admins authenticate via backend-issued HS256 JWTs signed with JWT_SECRET — they do not pass through Keycloak.

06

Vendor Portal

A Next.js 15 App Router project, but effectively a single-page app: all UI lives in app/page.tsx as one client component with internal view state (login, onboarding, dashboard, adminLogin, adminDashboard). No separate API client — inline fetch() against NEXT_PUBLIC_API_URL (default http://localhost:4000).

It stores the backend-issued JWT in React state and sends Authorization: Bearer to /vendor/onboarding, /vendor/login, /vendor/admin/login, /vendor/admin/summary, /vendor/shops/:id/summary, /vendor/shops/:id/products, and /vendor/orders/:id/status. Deployed via NIXPACKS (next build → next start, healthcheck /).

Keycloak

Keycloak 25 runs from infra/keycloak with a custom Dockerfile that imports the bazzato realm on boot. docker-entrypoint.sh parses DATABASE_URL into KC_DB_*, configures edge-proxy for Railway (KC_PROXY=edge, hostname from RAILWAY_PUBLIC_DOMAIN), then runs kc.sh build and kc.sh start --import-realm --optimized. Dual Authentication PathsFIG. 2

Customer auth

Handled by src/security/keycloak.ts via requireAuth on /orders. With KEYCLOAK_ISSUER + KEYCLOAK_JWKS_URL set, Bearer tokens validate against Keycloak's JWKS (issuer + audience). Non-prod has two dev fallbacks: no config → development-user; a mock-customer-{phone} token → customer-{phone}. Production disables both and requires Keycloak.

Mock OTP

src/services/otpService.ts: with OTP_PROVIDER=mock the code is fixed to MOCK_OTP_CODE (default 1234), stored in memory. With =sms, a random 4-digit code is delivered via console or HTTP. Verify returns a mock-customer-{phone} token (dev only).

Vendor / admin auth

src/security/appToken.ts issues HS256 JWTs signed with JWT_SECRET (12h), payload { phone, role, shopId? }. requireAppRole("vendor"|"admin") guards routes; admins reach any shop, vendors are pinned to shopId via requireSameShop. Vendors never touch Keycloak.

Order Lifecycle

  1. Customer browses catalog via public GET /catalog/*.
  2. Creates an order — POST /orders with Bearer token and { phone, shopId, items, deliveryAddress }.
  3. Backend computes subtotal + DEFAULT_DELIVERY_FEE, creates the order at status placed, records a timeline entry.
  4. Customer subscribes to WS /orders/:id/live?token= (15s HTTP poll fallback).
  5. Vendor advances status — PATCH /vendor/orders/:id/status.
  6. Backend emits orderEvents.orderUpdated → WebSocket pushes order.updated to the customer.

Progression: placed → accepted → preparing → ready → completed (or rejected / cancelled). scripts/smoke-holistic.mjs exercises the full loop end-to-end.

Deployment

Railway project grand-analysis hosts two environments mapped to Git branches: develop (staging) and master (production). Each runs the full stack plus its own databases.

ServiceBuilderStartHealthcheckRoot
backendNIXPACKSdb:deploy && start/healthapps/backend
vendorNIXPACKSnext start/apps/vendor
mobileDockerfileserve dist/apps/mobile
keycloakDockerfilekc.sh start --import-realm/realms/bazzatoinfra/keycloak
Monorepo gotcha Each service's rootDirectory must be set on its Railway service instance (e.g. apps/backend). Without it the builder runs from the repo root, never reads the app's railway.toml/Dockerfile, and fails with “No start command detected.” This setting lives in Railway's service config — not in git — so it is not committed.

Both environments default to mock OTP: backend OTP_PROVIDER=mock / MOCK_OTP_CODE=1234, mobile EXPO_PUBLIC_AUTH_PROVIDER=mock-otp; backend CORS_ORIGIN lists that env's vendor and mobile URLs.

14

Notable Details

  • Dual auth — customers via Keycloak/mock OTP; vendors & admins via backend HS256 JWTs.
  • Dual data source — memory is the zero-setup default; Postgres is opt-in via DATA_SOURCE=postgres + migrations.
  • Memory-mode quirk — the in-memory catalog's getShopByOwnerPhone() ignores the phone and returns the first shop.
  • Shared package unused — @bazzato/shared is scaffolded but not imported; each app defines its own types.
  • Vendor is a single-file SPA — all views in one Next.js page with view-state routing.
  • Mobile uses React Navigation, not expo-router — 11 screens wired in App.tsx.
  • Production guardrails — JWT_SECRET must not be default, and Keycloak is required (dev fallbacks off) when NODE_ENV=production.
